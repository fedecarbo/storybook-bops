# Check Constraints — User Journey

The case officer reviews planning constraints that affect the application site. Constraints are automatically identified by **PlanX** and **Planning Data** when the application is submitted — these include things like Conservation Areas, Listed Buildings, Tree Preservation Orders, and Flood Zones. The officer checks these are correct, adds any missing constraints manually, and marks the task as complete.

This is an **internal task** — there is no email or external actor involvement. It typically takes a few minutes unless the constraint data needs significant correction.

A link to "View map on Planning Data" opens a map showing all constraint geometries overlaid on the site.

If the officer knows of a constraint that wasn't automatically identified, they expand the **"Add constraints"** details panel and search by name or category. The search results show available constraints (those not already on the application) with an **"Add"** link.

After clicking "Add", the constraint appears in the identified constraints table. Unlike PlanX-identified constraints (which show `-` in the Action column), manually-added constraints show a **"Remove"** link so the officer can undo the addition if needed. The source column shows the officer's name instead of "PlanX".

Once the officer is satisfied that all relevant constraints are accounted for, they click **"Save and mark as complete"**. The task status changes to **Complete** (green tag) in the task list sidebar. An audit entry "constraints_checked" is recorded against the application.
