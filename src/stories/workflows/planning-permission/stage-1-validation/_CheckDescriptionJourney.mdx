# Check Description — User Journey

This documents the end-to-end journey of checking whether the application description matches the development shown in the plans. It involves **three actors**: the **case officer** (in BOPS), the **system** (automatic emails), and the **applicant** (via a secure response link).

The description check has two main paths: the officer confirms the description is correct (quick path), or proposes an amended description. The amended description path further branches depending on whether applicant agreement is required.

**Who:** Case officer
**Where:** Check description form (sub-radio)

The officer has two options:

**"Yes, applicant agreement needed"** — The amended description will be sent to the applicant for approval. They have 5 business days to respond. If they don't respond, the change is automatically accepted to avoid delaying the application.

<Canvas of={CheckDescriptionStories.SelectedNoWithApproval} />

**"No, update description immediately"** — The description is updated right away. The applicant receives a notification email but doesn't need to approve it. This is used for minor corrections (standardising terminology, fixing typos).

The task is complete when:
- The officer confirmed the description was correct (selected "Yes" in step 1), or
- The applicant approved the amended description and the officer saved, or
- The request auto-closed after 5 days with no response, or
- The officer chose "update immediately" (no applicant approval needed)
