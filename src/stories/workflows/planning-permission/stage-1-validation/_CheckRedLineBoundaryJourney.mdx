# Check Red Line Boundary — User Journey

This documents the end-to-end journey of checking the applicant's red line boundary (site outline). It involves **three actors**: the **case officer** (in BOPS), the **system** (emails), and the **applicant** (via a response link). The case officer checks whether the submitted boundary is correct. If not, they propose a change which the applicant must agree or disagree with.

**Who:** Case officer
**Where:** Validation task list → Check red line boundary

The case officer opens the task and sees the applicant's submitted red line boundary displayed on a map. They are asked: *"Is the red line boundary correct?"*

- If **Yes** → the task is saved and marked as complete. Journey ends.
- If **No** → the officer is taken to a form to propose a new boundary.

**Where:** Propose red line boundary change form

If the officer selects "No", they are taken to the propose change form. Here they:

1. See the applicant's existing boundary on the map (shown in grey)
2. Draw the proposed new boundary on the map (shown with a red dashed line)
3. Write a reason explaining why the boundary needs to change
4. Click "Send request" to notify the applicant

**Who:** System (automatic)
**To:** The applicant's email address

Once the officer submits the proposed change, an email is sent to the applicant with:
- The application reference and site address
- A link to review the proposed boundary change and respond
- A deadline — if the applicant doesn't respond within 5 business days, the change is automatically approved

- A radio button choice: **"Yes, I agree"** or **"No, I disagree"**

**Who:** Case officer
**Where:** Validation task list

The task is complete when either:
- The officer confirmed the original boundary was correct (selected "Yes" in step 1)
- The applicant approved the proposed change and the officer saved (step 6a)
- The request auto-closed after 5 business days and the officer saved

The task status changes to **Complete** with a green tag in the validation task list.
