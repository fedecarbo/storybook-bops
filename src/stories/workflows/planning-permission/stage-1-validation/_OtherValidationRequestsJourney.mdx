# Other Validation Requests — User Journey

This documents the end-to-end journey of flagging generic validation issues and requesting the applicant to resolve them. It involves **three actors**: the **case officer** (in BOPS), the **system** (automatic emails), and the **applicant** (responding via a secure link).

"Other validation requests" is the catch-all mechanism for any validation issue not covered by specific request types (documents, description, red line boundary, fee, ownership certificate). The officer explains **why** the application is invalid and **how** the applicant can fix it. The applicant responds with free text.

The officer clicks "Add another validation request" and is taken to a form with two fields:
1. **Reason** — tell the applicant why the application is invalid
2. **Suggestion** — explain how the applicant can make the application valid

The request is saved but **not sent** until the application is marked as invalid (or immediately if the application is already invalidated).

**Who:** System (automatic)
**To:** Applicant email (and agent if applicable)

When the validation requests are sent (either immediately when created during an invalidated state, or batched as part of the validation decision), the applicant receives an email. The email tells them that the case officer needs more information and provides a secure link to view and respond to the requests.

If the applicant does not respond within **15 business days**, the application may be returned with a refund.
