# Expired Token Journey — Current vs Ideal

This documents the user journey when a **password reset** or **email confirmation** link has expired (after 6 hours). It covers two scenarios: a **confirmed user** resetting their password, and an **unconfirmed user** setting up their account for the first time.

Both flows use the same email template and the same reset password token mechanism (via the `EmailConfirmable` concern), so the expired token experience is identical for both — which is part of the problem.

## Summary of Changes Needed

| Issue | Current | Ideal |
|-------|---------|-------|
| **When is the user warned?** | After filling in and submitting the form | Immediately on clicking the expired link |
| **Error message (confirmed user)** | "The reset password link you used no longer works. Please request a new link and try again." | "Your password reset link has expired. Request a new one using the 'Forgot your password?' link below." |
| **Error message (unconfirmed user)** | Same generic message as above | "Your account confirmation link has expired. Contact your administrator to request a new one." |
| **Recovery path clear?** | No — user has to figure it out | Yes — message includes specific next step |

### Technical implementation required

1. **Create `Users::PasswordsController`** — override the `edit` action to validate the token before rendering the form
2. **Differentiate by user status** — check `confirmed?` to show the appropriate message
3. **Update locale strings** — add context-specific expiry messages to `config/locales/devise.en.yml`
4. **Update routes** — add `passwords: "users/passwords"` to the `devise_for` controllers hash
