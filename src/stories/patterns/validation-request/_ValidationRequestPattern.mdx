# Validation request

A structured request-response cycle where case officers ask applicants for information or agreement during the validation stage of a planning application. All 10 request types share a common lifecycle and UI skeleton with 3 response mechanisms.

<Canvas of={ValidationRequestStories.LifecycleOverview} />

<hr />

## Why this pattern exists

Validation requests are the primary mechanism for officer-applicant communication during the validation stage. When a submitted application is incomplete or needs changes, the case officer creates one or more validation requests — each targeting a specific issue.

There are **10 distinct request types** in the system, but they all follow the same lifecycle and fall into just **3 response mechanisms**. This pattern documents the shared skeleton so that:

- New request types can be designed consistently
- Existing types can be improved against a documented baseline
- Designers can understand the common structure without reading 10 separate implementations

<hr />

## When to use this pattern

- When building any new validation request type
- When the interaction requires the officer to send a structured request to the applicant and receive a structured response
- When the applicant's response follows one of the 3 mechanisms (agree/disagree, text, document upload)

## When not to use this pattern

- For officer-to-officer communication (use internal notes or assignments instead)
- For automated system notifications that don't require a response
- For post-decision communication with applicants

## System notification

When a request is sent, the system emails the applicant with a secure link to respond. All types use the same email template structure — only the subject line and body details vary by type.

<Canvas of={ValidationRequestStories.EmailNotification} />

<hr />

## Applicant landing page

After clicking the email link, the applicant sees a landing page listing all open requests as a task list. This page aggregates requests across all types — the applicant works through them one by one.
